Semantic type system¶

Contents

  • Semantic type system
    • Type
    • Statement
    • Type system
    • Examples

The semantics module implements a basic description language for describing how types relate to each other (a "type system").

Type¶

The base object in this system is a Type:

class data_api.semantics.Type(spec)[source]¶

An object describing a single type.

__init__(self, spec)[source]¶

Create type from specification.

Parameters:spec (dict) -- Specification, see below

The specification, a dict that could have been parsed directly from a JSON string (so, a dict containing only lists, dicts, and JSON scalars):

{
// Optional namespace for type name
"namespace": "kbase.us",
// Type name using UpperCamelCase convention
"name": "BananaBunch",
 // Description of what these objects mean
"description": "A yellow fruit",
// List of properties, each with name and type.
// These are the value(s) for your type.
"properties": [
    {"name": "num", "type": "int"},
    {"name": "color", "type": "string"},
],
// Zero or more statements about the relationship to
// other types, which are pairs of the form
// ["relationship", "NameOfType"].
// The Statements class defines constants, given
// in comments, for standard relationships.
// Arbitrary relationships are also possible.
"statements": [
    ["isSubclassOf", "Fruit"],     // .extends
    ["aggregates", "Banana"],      // .agg
    ["derivedFrom", "BananaTree"], // .derived_from
    ["inAFruitBasketWith", "GrapeBunch"],
}

Statement¶

The statements that form the inter-type relationships can use, as their predicate, the constants defined in the Statement class:

class data_api.semantics.Statement(class1, relationship, class2)[source]¶

Single RDF-like triple of the form (subject, predicate, object), where the 'subject' and 'object' are both Type and the predicate is a string desribing the nature of the association.

agg = 'aggregates'¶

Multiple objects are aggregated in the subject

derived_from = 'derivedFrom'¶

The object is derived from the subject

extends = 'isSubclassOf'¶

The subject is a sub-class of the object

Type system¶

To parse and validate Type statements, and output a coherent joint schema, there needs to be a class that can know about multiple types at once. This class is called TypeSystem:

class data_api.semantics.TypeSystem(namespace='', version=None)[source]¶

Multiple types, that may refer to each other by name.

Importantly, versions are a property of this class, and not of individual types.

__init__(self, spec)[source]¶

Initialize an empty container.

Parameters:namespace (str) -- optional default namespace to prepend to the referenced names, unless they are already qualified

You can add all your types with the TypeSystem.add() method:

TypeSystem.add(self, type_)[source]¶

Add a type.

Parameters:type (Type) -- New type to add
Returns:None

Then, once you have added all your types, you can get serialized schema that combine all the types in a way that's palatable to a given format. Right now, the only supported format is Avro:

Create and return avro schema.

Parameters:name -- Name of the schema to create, which should match the name of some type given to add().
Returns:The new avro schema.
